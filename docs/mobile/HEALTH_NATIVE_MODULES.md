# Health Native Modules ‚Äî HealthKit, Health Connect & Samsung Health

This document covers the on-device native module configuration serialized by **PR-HK-CFG**. It is the single source of truth for how The Growth Project reads (and selectively writes) wearable + health data on iOS and Android, how to produce a dev-client build that includes these native modules, and what every permission is for so QA can verify the consent flows.

> **Scope note.** PR-HK-CFG is **strictly additive configuration only**. It installs the native bridges and declares permissions/plugins. The actual on-device connector logic lands in the follow-on PRs:
> - **PR-HK-2.a** ‚Äî Apple HealthKit connector (iOS)
> - **PR-HK-2.b** ‚Äî Health Connect / Google Fit connector (Android)
> - **PR-HK-2.c** ‚Äî Samsung Health connector (Android)
>
> Until those land there is **no behavioral change** in the app ‚Äî the modules are present and permissions are declared, but nothing requests consent or reads data yet.

## 1. Which provider uses which native module

| Provider | Platform | Native module | npm package | Pinned version |
| --- | --- | --- | --- | --- |
| **Apple HealthKit** | iOS | `react-native-health` (HealthKit bridge + Expo config plugin) | [`react-native-health`](https://www.npmjs.com/package/react-native-health) | `1.19.0` |
| **Health Connect** (also the modern path for Google Fit data) | Android | `react-native-health-connect` (Health Connect Jetpack client bridge + Expo config plugin) | [`react-native-health-connect`](https://www.npmjs.com/package/react-native-health-connect) | `3.5.3` |
| **Samsung Health** | Android | Reads through **Health Connect** (Samsung Health writes into Health Connect on supported devices) + Samsung Sensor SDK additional-data permission | shares `react-native-health-connect` + Samsung permission declarations | n/a (no separate npm bridge in this PR) |

**Why Samsung Health rides on Health Connect.** On modern Samsung devices, Samsung Health syncs its data into Android Health Connect, and Health Connect exposes a standardized permission model using `android.permission.health.*` strings (see the [Health Connect data types & permissions reference](https://developer.android.com/health-and-fitness/guides/health-services/permissions)). The Samsung-specific read path additionally needs the Samsung Sensor SDK permission `com.samsung.android.hardware.sensormanager.permission.READ_ADDITIONAL_HEALTH_DATA` for on-watch sensor reads (see the [Samsung Health Sensor SDK permission request guide](https://developer.samsung.com/health/sensor/guide/permission-request.html)). Both are declared in this PR so the 2.c connector has everything it needs without another config PR.

### Expo config plugins (under `expo.plugins`)

```jsonc
[
  "react-native-health",
  {
    "healthSharePermission": "‚Ä¶Apple Health read description‚Ä¶",
    "healthUpdatePermission": "‚Ä¶Apple Health write description‚Ä¶"
  }
],
"react-native-health-connect"
```

- The `react-native-health` plugin injects `NSHealthShareUsageDescription` / `NSHealthUpdateUsageDescription` into the iOS Info.plist and adds the `com.apple.developer.healthkit` entitlement automatically during prebuild / EAS Build.
- The `react-native-health-connect` plugin adds the Health Connect permissions-rationale intent filter to the Android manifest. It takes no options.

## 3. Permissions explainer (for QA)

### iOS ‚Äî `expo.ios.infoPlist`

| Key | Purpose | What QA should see |
| --- | --- | --- |
| `NSHealthShareUsageDescription` | Read access to Apple Health (sleep, heart rate, workouts, etc.). | Shown on the HealthKit read-consent sheet when the HealthKit connector requests access. |
| `NSHealthUpdateUsageDescription` | Write access ‚Äî lets the app save logged workouts back to Apple Health. | Shown on the HealthKit write-consent sheet. |
| `NSMotionUsageDescription` | Motion & step-count access used by HealthKit step queries. | Shown when step/motion data is first requested. |

> HealthKit consent in iOS is per-data-type and is requested by the **2.a** connector, not by config alone. The HealthKit capability/entitlement is added automatically by the config plugin during the native build.

### Android ‚Äî `expo.android.permissions`

**Health Connect read permissions** (one per data type the coaching model uses):

| Permission | Data |
| --- | --- |
| `android.permission.health.READ_STEPS` | Step count |
| `android.permission.health.READ_HEART_RATE` | Heart rate |
| `android.permission.health.READ_SLEEP` | Sleep sessions |
| `android.permission.health.READ_ACTIVE_CALORIES_BURNED` | Active calories |
| `android.permission.health.READ_TOTAL_CALORIES_BURNED` | Total calories |
| `android.permission.health.READ_DISTANCE` | Distance |
| `android.permission.health.READ_EXERCISE` | Exercise / workout sessions |
| `android.permission.health.READ_OXYGEN_SATURATION` | SpO‚ÇÇ |
| `android.permission.health.READ_RESPIRATORY_RATE` | Respiratory rate |
| `android.permission.health.READ_HEART_RATE_VARIABILITY` | HRV |
| `android.permission.health.READ_BODY_TEMPERATURE` | Body temperature |
| `android.permission.health.READ_WEIGHT` | Weight |
| `android.permission.health.READ_BODY_FAT` | Body fat % |
| `android.permission.health.READ_VO2_MAX` | VO‚ÇÇ max |
| `android.permission.health.READ_BASAL_BODY_TEMPERATURE` | Basal body temperature |
| `android.permission.health.READ_BLOOD_PRESSURE` | Blood pressure |
| `android.permission.health.READ_HEALTH_DATA_IN_BACKGROUND` | Allows background reads (subject to user opt-in in Health Connect settings). |

**Activity / Samsung permissions:**

| Permission | Purpose |
| --- | --- |
| `android.permission.ACTIVITY_RECOGNITION` | Required for step/activity data on Android. |
| `com.samsung.android.hardware.sensormanager.permission.READ_ADDITIONAL_HEALTH_DATA` | Samsung Sensor SDK additional health-data reads (BIA/PPG/etc. paths used by the Samsung Health connector). |

> On Android, the user grants Health Connect permissions in the **Health Connect** system UI (or the in-app rationale flow), not the standard runtime-permission dialog. QA should verify the app appears under **Health Connect ‚Üí App permissions** after the 2.b connector requests access. Permission strings declared here are valid against the Health Connect permission model documented by [Android Health Connect](https://developer.android.com/health-and-fitness/guides/health-services/permissions).

## 4. Dev-client build instructions

These native modules are **not** available in Expo Go. You must build and install a **development client**.

### One-time prerequisites
- EAS CLI (`eas-cli >= 18.0.0`, see `eas.json` ‚Üí `cli.version`).
- Logged in: `eas login`.
- For iOS device builds: Apple Developer account credentials (EAS manages certificates).

### Build a dev client

```bash
# Android dev client (APK, internal distribution)
eas build --profile development --platform android

# iOS dev client (device or simulator)
eas build --profile development --platform ios
```

Install the resulting build on a device/emulator, then start Metro:

```bash
npx expo start --dev-client
```

### Local prebuild sanity check (optional, do NOT commit native folders)

```bash
npx expo prebuild --no-install --clean=false
```

This generates `ios/` and `android/` so you can inspect the merged `Info.plist`, `*.entitlements`, and `AndroidManifest.xml`. **These folders are git-ignored and must never be committed** ‚Äî The Growth Project uses the Expo **managed** workflow, and EAS Build regenerates native projects on every build via the `prebuildCommand` configured in `eas.json`.

After inspecting, remove them:

```bash
rm -rf ios android
```

### Verifying the config plugins applied

After a prebuild, confirm:

```bash
# iOS ‚Äî health usage strings present
grep NSHealth ios/*/Info.plist

# iOS ‚Äî HealthKit entitlement present
grep healthkit ios/*/*.entitlements

# Android ‚Äî health permissions present
grep "permission.health" android/app/src/main/AndroidManifest.xml
```

## 5. Notes for the connector PRs (2.a / 2.b / 2.c)

- **Do not** add native config here ‚Äî the mutex for mobile config is owned by PR-HK-CFG. Connector PRs consume what is declared here.
- **HealthKit (2.a):** request per-type read/write permission at runtime via `react-native-health`. Entitlement and Info.plist strings are already in place.
- **Health Connect (2.b):** check Health Connect availability, then request the subset of declared `android.permission.health.*` permissions actually used by a given screen (Health Connect requires least-privilege requests).
- **Samsung Health (2.c):** prefer the Health Connect path for synced data; use the Samsung Sensor SDK permission only for on-watch sensor reads.
